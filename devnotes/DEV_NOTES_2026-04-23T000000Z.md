# DEV NOTES — 2026-04-23T00:00:00Z

目的
- 紀錄關於 atlas 的治理定位（archive vs governance/index）與後續分 repo 策略決策與實作要點。
- 提供可直接採用的 catalog 與 workflow 思路，讓 active/social repo（會與外部互動的）與 atlas（治理/檔案式治理）分工明確。

決定性建議（結論）
- atlas 保持為治理 / 檔案式治理（archive + canonical index + policy templates）。
- 會頻繁與社群互動或需要外部發布/issue 流程的專案（例如 prompts、latex-templates、social-promos 等）應該拆成獨立 repo。
- 在 atlas 中維護 machine-readable catalog（catalog.yml 或 catalog.json）與 README（human-readable index），作為使用者與 automation 的單一入口。

為什麼這樣（摘要）
- 分離關注點：archive 與 active governance 的需求不同（權限、CI、Secrets）。
- 權限管理更清晰：active repo 可放發佈 token、org-level secrets、公開 issues；atlas 則保持受控、只允許維運人員修改。
- discoverability：外部使用者先看 atlas 的目錄頁或 GH Pages，就能直接被導到對應 active repo，避免多層搜尋。

- sinwulok/atlas (治理 / index / archive)
  - README.md（快速目錄 + policy quickstart）
  - catalog.yml（機器可讀）
  - governance/（policy、MCP 規範、CI templates）

- sinwulok/prompts (active, social)

同步與自動化模式（兩種主流選擇）
- 子 repo -> atlas push（子 repo 主動更新 catalog）
  - 優點：變更即時、簡單
  - 需求：子 repo 需有一個最小權限 PAT 或使用 GitHub App（建議使用 GitHub App）來更新 atlas。
  - 風險：token 管理須嚴格（最小權限、定期旋轉）。
- atlas pull（atlas 主動擷取子 repo metadata）
  - 優點：較安全，不需授權子 repo 寫 atlas；管理集中
  - 實作：atlas 的 workflow 週期性呼叫 GitHub API 讀取子 repo metadata、更新 catalog.yml。
  - 建議：預設使用 atlas pull 模式，啟用需要頻繁更新的再考慮 push 模式或 GitHub App。

治理 / gating（合併門檻）建議
- 把 policy-check workflow 加入 child repo 的 PR 流程：
  - child repo 在 PR 時 clone atlas（或呼叫 atlas 的 policy-check action），根據 governance/rules 執行驗證（metadata 格式、license、MCP checks 等）。
  - 將該 workflow 設為 branch protection 的 required status check。
- CODEOWNERS 用於指定審核者。
- Secrets（例如發佈到 X 的 TOKEN、Overleaf token）放在需要執行發佈的 child repo，atlas 不放任何發佈 credentials。

Prompts / Social repo 的特殊考量
- Prompts 會頻繁發布到 X，可能產生很多 issue/PR/threads —— 建議拆出獨立 repo：
  - 方便置放 publish tokens、設置發佈 workflow、維護 issue-based editorial 流程。
- 在 prompts repo README 明確標註「Governed by sinwulok/atlas」並連結到 atlas 的 policy 與 catalog entry。
- 若需要在 atlas 顯示 social 活動狀態，可在 child repo 中定期更新 metadata（例如 last_posted_at、engagement 指標）並由 atlas pull。

安全建議
- 儘量使用 GitHub App 或 Fine-grained PAT（限定權限並設定 expiry）來做跨 repo 寫入；避免放 long-lived 全權 token。  
- atlas 不存發佈 token；只存 policy 與 metadata。  
- 對敏感操作（score runs、publish）加上 audit logs（GitHub App + actions log）與最小授權原則。
